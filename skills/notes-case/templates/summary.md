# <case 标题> — 调试总结

> SoC/板型: <…>   ·   SDK 基线: <…>   ·   完成日期: YYYY-MM-DD
> 关联 case: <SDK>/done/<年月>/<case目录>   ·   类型: bug | feature

## TL;DR
> 一两句话:什么问题、根因是什么、怎么解决的。让人三秒抓到要点。

## 问题
> 从 case 现象段提炼:现象 + 影响 + 复现条件/概率。

## 根因
> 从 findings 提炼:为什么会这样,讲清机理(不只是"改了哪行",而是"为什么是它")。

## 解决方案
> 从 patch / decisions 提炼:改了什么、为什么这么改、落在哪个子仓库@commit。
> 若有备选方案和取舍,一并写明(便于他人判断是否适用自己的场景)。

## 验证
> 从 verify 提炼:怎么确认修好了(步骤 + 结果),覆盖了哪些边界、还有哪些没覆盖。

## 避坑经验 / 教训
> 这一段最值钱,面向"下次更快":
- 排查时走过的弯路、agent 一度带偏的点
- 更快定位的线索/命令(早知道就好了的那条)
- 适用范围:同类问题(其它板/芯片/驱动)能否借鉴,边界在哪
